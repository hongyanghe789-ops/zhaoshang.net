### 1.3 User Link App 定位与架构

#### 1.3.1 产品定义

User Link（简称UL）是公司级的**商务资源社交App**，在本项目中定位为**辅助连接层**和**小程序宿主**。旗舰版的移动端管理和租客端功能以小程序形式嵌入UL App运行，共享UL的IM、通讯、支付等底层能力。

| 维度 | 定义 |
|------|------|
| **产品名称** | User Link |
| **产品类型** | 商务资源社交连接App + 小程序宿主平台 |
| **核心定位** | 汇聚公司自有平台资源，让用户基于资源与对方进行连接 |
| **在本项目中的角色** | 辅助层 — 为旗舰版提供移动端触达（小程序宿主）和社交连接（IM/电话） |
| **核心原则** | UL只管展示和连接，业务逻辑由各平台（旗舰版等）自己控制 |
| **小程序能力** | 旗舰版移动端管理小程序 + 租客端小程序，均以UL为宿主App运行 |
| **沟通方式** | 在线IM沟通 + 电话虚拟号对接 |
| **资源来源** | 招商网络平台、星斗云（律师）平台、未来更多平台 |

#### 1.3.3 UL与各平台的职责边界

| 职责 | User Link | 各业务平台（招商网络/星斗云等） | 旗舰版（本项目核心） |
|------|-----------|-------------------------------|-------------------|
| 资源展示 | 统一卡片展示、搜索、推荐 | Push资源数据到UL | 通过平台间接Push |
| 连接发起 | 提供IM/虚拟号拨打入口 | 校验用户是否有权发起连接 | 接收对接回调，转为线索 |
| 沟通过程 | 承载IM消息/通话 | 不介入沟通过程 | 不介入，接收回调数据 |
| 业务转化 | 不处理业务逻辑 | 接收UL回调，完成业务闭环 | 全部业务逻辑在旗舰版处理 |
| 数据归属 | 沟通记录归UL | 平台侧数据归平台 | 业务数据归旗舰版（CRM/合同/财务等） |
| 权益消耗 | 发起对接前请求平台校验 | 平台自行扣减权益 | 通过会员管理控制权益 |
| 小程序宿主 | 提供小程序运行容器+底层能力 | — | 提供移动端管理+租客端小程序的业务逻辑 |
| 用户体系 | 统一账号（公司级SSO） | 各平台用户通过SSO关联 | 旗舰版用户通过SSO关联 |

                             用户发起法律咨询
                                    │

旗舰版 → 载体发布 ──────→ 招商网络平台 ──→ Push到UL

旗舰版 ← 载体反馈数据 ←── 招商网络平台 ←── UL曝光/点击/对接数据

旗舰版(移动端管理小程序) ──→ 以UL小程序运行 ──→ 复用UL IM/推送能力

旗舰版(运营管理) ←────────── 租客端小程序 ←──── 报修/预约/评价等数据
```

### 1.5 四大核心价值层次（升级版）

```
第一层（基础价值）★旗舰版核心：替代Excel管房源、管合同、管账单
                    → 功能完整度高，全模块覆盖全业务链

第二层（效率价值）★旗舰版核心：公域获客+私域跟进+租后运营一体化闭环
                    → 核心差异化优势，竞品难以复制

第三层（决策价值）★旗舰版核心：数据看板+AI分析辅助经营决策
                    → 管理层购买决策的关键

第四层（生态价值）UL辅助：User Link跨平台资源社交连接
                    → 打通公司全平台资源，形成商务社交生态壁垒
                    → UL同时作为旗舰版移动端/租客端小程序的宿主
```

## 四、ABAC权限模型（全新改造）

### 4.1 从RBAC到ABAC的改造说明

**改造原因**：
- 原RBAC模型（角色+功能权限+数据权限）在多园区集团管控、跨部门协作、精细化字段级控制等场景下灵活性不足
- ABAC（Attribute-Based Access Control）基于属性的访问控制，可动态组合主体属性、资源属性、环境属性、操作属性进行策略决策，支撑更复杂的业务场景

### 4.2 ABAC四维属性模型

```
┌─────────────────────────────────────────────────────────────┐
│                    ABAC策略决策引擎（PDP）                    │
│                                                             │
│   访问请求 = f(主体属性, 资源属性, 操作属性, 环境属性)        │
│                                                             │
│   策略规则：IF 条件组合 THEN Permit/Deny                     │
└─────────────────────────────────────────────────────────────┘

┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  主体属性     │  │  资源属性     │  │  操作属性     │  │  环境属性     │
│  (Subject)   │  │  (Resource)  │  │  (Action)    │  │ (Environment)│
│              │  │              │  │              │  │              │
│ · 用户ID     │  │ · 资源类型    │  │ · 操作类型    │  │ · 访问时间    │
│ · 部门       │  │  （合同/房源  │  │  （查看/编辑  │  │ · 访问IP     │
│ · 职位/职级   │  │   /线索/...）│  │   /删除/导出  │  │ · 设备类型    │
│ · 角色标签    │  │ · 所属园区    │  │   /审批）     │  │ · 地理位置    │
│ · 入职年限    │  │ · 所属部门    │  │              │  │ · 风险等级    │
│ · 管辖园区    │  │ · 创建人     │  │              │  │              │
│ · 技能标签    │  │ · 负责人     │  │              │  │              │
│ · 所属集团    │  │ · 数据密级    │  │              │  │              │
│              │  │ · 金额区间    │  │              │  │              │
│              │  │ · 客商类型    │  │              │  │              │
└──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘
```

### 4.3 ABAC策略规则示例

| 策略ID | 策略名称 | 规则描述 | 效果 |
|--------|---------|---------|------|
| P001 | 部门数据隔离 | Subject.部门 == Resource.所属部门 | Permit |
| P002 | 集团总部全局可见 | Subject.角色标签 CONTAINS "集团管理员" | Permit（跨园区） |
| P003 | 大额合同审批 | Resource.类型=="合同" AND Resource.金额>100万 AND Action=="审批" → 需总经理节点 | 路由 |
| P004 | 敏感数据保护 | Resource.数据密级=="高" AND Subject.职级<"总监" → 字段脱敏 | 脱敏 |
| P005 | 非工作时间导出限制 | Action=="导出" AND Environment.时间 NOT IN 工作时段 AND 导出量>100条 → 熔断+通知超管 | Deny+报警 |
| P006 | 多园区管辖 | Subject.管辖园区 CONTAINS Resource.所属园区 | Permit |
| P007 | 客商CRM跨部门共享 | Resource.类型=="客商" AND Resource.客商类型=="潜在客户" AND Subject.角色标签 CONTAINS "招商人员" | Permit（只读） |
| P008 | 移动端位置校验 | Action=="抄表" AND Environment.设备=="移动端" AND GPS距离>500米 | Deny |

### 4.4 ABAC与原RBAC的兼容策略

为保证平滑过渡，ABAC系统兼容原有RBAC概念：

```
原RBAC概念        ABAC映射方式
──────────      ──────────────────
角色             → 角色 = 一组预设的主体属性标签集合
                   例："招商经理"角色 = {角色标签:"招商管理", 职级:"经理"}

功能权限          → Action属性 + Resource.类型 的策略规则
                   例：角色标签:"招商管理" + Action:"编辑" + Resource:"线索" → Permit

数据权限（本人）   → Subject.用户ID == Resource.创建人/负责人
数据权限（本部门） → Subject.部门 == Resource.所属部门
数据权限（全部）   → 无资源属性过滤
数据权限（指定）   → Subject.管辖部门 CONTAINS Resource.所属部门

字段权限          → Resource.字段名 + Subject属性 的策略规则
                   例：字段"合同底价" + Subject.职级<"总监" → Hidden
```

│        │ 策略管理 │  │ 属性查询 │  │ 策略存储 │      │
│        │ （管理端 │  │ （实时获 │  │ （策略库 │      │
│        │ 配置UI）│  │ 取属性） │  │ 缓存）  │      │
│        └─────────┘  └─────────┘  └─────────┘      │
└────────────────────────────────────────────────────┘

PEP = Policy Enforcement Point（策略执行点，嵌入API网关）

## 七、全局业务规则

### 7.1 载体层级规则

```
园区（Park）
 └── 建筑（Building）
      └── 楼层（Floor）
           └── 房源（Room/Unit）← 最小可出租单元
```

- 一个园区下可包含N个建筑
- 一个建筑下包含N个楼层
- 一个楼层下包含N个房源
- 房源是合同绑定的最小颗粒度
- **新增**：支持一房多租（孵化器工位场景）和一客多房（多楼层合同）

### 7.2 房源状态机（扩展版）

```
[空置] ──→ [即将空置（可预订）] ──→ [锁定/谈判中] ──→ [在租]
  ↑                                      │              │
  │                                  审批拒绝           │
  │                                  释放回空置         合同退租/到期
  │                                                     │
  └──────────────────────────────────────────────────────┘

※ 合同进入审批中 → 房源自动锁定 → 平台状态实时同步
※ 合同生效 → 房源在租 → 平台自动下架 → 生成成交喜报
※ 退租流程触发 → 自动标记"即将空出" → Push到UL/平台
```

### 7.3 合同状态机

```
        ┌───────────────────────────────────────────────┐
        │                                               │
        ▼                                               │
    [待执行] ──→ [执行中] ──→ [已到期] ──→ [已退租]      │
        │           │                                   │
        │           ├──→ [变更中] ──→ [执行中]（变更后）  │
        │           │                                   │
        │           ├──→ [续租] ──→ 新合同[待执行]       │
        │           │                                   │
        │           └──→ [退租中] ──→ [已退租]           │
        │                                               │
        └──→ [已作废] ─────────────────────────────────┘
```

### 7.4 客商(CRM)生命周期规则

```
来源渠道                    客商状态               业务联动
──────────                ──────────             ──────────
IM/电话对接转化     ──→   [潜在客户] ──→ 关联线索/项目
UL App对接回调      ──→        │
大数据/工单/自主     ──→        ▼
                         [跟进中]
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
              [签约→租客]         [无效→关闭]
                    │
            ┌───────┴───────┐
            ▼               ▼
         [在租]          [退租/流失]
            │               │
            ▼               ▼
        [续约/扩租]    [流失客户池]
```

- 客商可同时拥有多个身份标签（如既是租客又是供应商）
- 潜在客户通过签约自动升级为租客，无需重新创建
- 客商详情页聚合展示：线索→项目→合同→账单全链路数据

### 7.5 线索→项目→签约流转规则

| 阶段 | 状态 | 转化条件 | 数据继承 |
|------|------|---------|---------|
| 线索创建 | 未跟进 | IM/电话/大数据/工单/自主/领导推荐/UL对接回调 | — |
| 线索跟进 | 跟进中 | 添加跟进日志 | — |
| 转为项目 | 已转项目 | 手动转化操作 | 线索信息回显；自动创建/关联客商（潜在客户） |
| 项目跟进 | 入库→洽谈中 | 推进项目进度 | — |
| 项目签约 | 已签约 | 创建关联合同 | 弹窗引导签约；客商身份升级为"租客"；平台携带信息自动填充 |
| 线索回收 | 已回收 | 长期未跟进/手动回收 | 进入回收站 |
| 线索关闭 | 已关闭 | 明确无效 | 进入回收站 |

### 7.6 账单生成规则（增强版）

- 合同签约后，系统根据合同计费规则自动生成账单
- 账单周期支持：月付/季付/半年付/年付/**按月数自定义**
- 递增规则支持：固定金额递增、固定比例递增、**按月数区间设置递增**
- 免租期处理：支持"冲抵租金"与"租期顺延"两种模式；"适用科目"可多选（免租不免管）
- 物业费/水电费按独立规则生成
- **虚拟合并账单**：同一租户同周期的多项费用通过"账单池"合并为单一多行明细账单
- **容错估算**：分表未抄录时基于历史均值估填，下月True-up修正
- 手动录入账单用于补充系统未覆盖的特殊费用

### 7.7 数据权限过滤规则（ABAC版）

所有列表查询均需经过ABAC策略引擎：

```
查询请求
    │
    ▼
PEP拦截 → 提取主体属性（用户/部门/角色标签/职级/管辖园区）
    │
    ▼
PDP策略匹配 → 遍历所有适用策略规则
    │
    ├── 匹配P001（部门隔离）→ 过滤资源.所属部门
    ├── 匹配P002（集团全局）→ 不过滤
    ├── 匹配P004（敏感脱敏）→ 特定字段脱敏
    ├── 匹配P006（多园区）  → 过滤资源.所属园区
    ├── 匹配P007（CRM共享）→ 潜在客户只读
    └── ...
    │
    ▼
返回过滤后的数据集 + 字段级脱敏/隐藏指令
```

## 十二、技术约定（供研发参考）

### 12.1 多租户隔离

- 数据层面：所有业务表以 `tenant_id`（单位ID）做租户隔离
- 应用层面：登录后全局注入租户上下文，所有查询自动附加租户过滤
- 集团管控场景：集团级视图可跨租户聚合查询（受ABAC策略控制）

### 12.2 ABAC技术实现

- PEP嵌入API网关层，所有请求统一拦截
- PDP作为独立微服务，策略评估结果缓存（TTL可配置）
- PAP提供管理后台UI，支持策略可视化编辑
- PIP对接组织架构/用户属性/资源元数据，实时查询
- 策略存储支持版本管理和灰度发布

### 12.3 软删除

- 全局采用软删除策略，删除操作标记 `is_deleted = true`
- 回收站功能基于软删除实现（招商管理已实现）

### 12.4 操作日志

- 所有关键业务操作记录操作日志（通用日志组件）
- 日志字段：操作人、操作时间、操作类型、操作内容、变更前后值
- **项目快照**：基于全量日志实现"时光机"功能，支持任意时间点状态回溯

### 12.5 API设计约定

- RESTful风格
- 列表接口统一支持：分页(page/pageSize)、排序(sortField/sortOrder)、筛选(filter参数)
- 响应格式统一：`{ code, message, data: { list, total, page, pageSize } }`
- UL资源Push接口遵循统一Push协议（见1.3.4节）

### 12.6 实时状态同步

- 合同状态变更 → 房源状态 → 平台房源状态：采用事件驱动（Event-Driven）架构
- 延迟要求：状态同步延迟 < 3秒
- 失败重试：最多3次，失败后进入人工处理队列
